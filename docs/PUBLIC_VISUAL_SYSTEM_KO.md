# RuleLink 공개 웹 시각 체계

## 1. 방향

RuleLink의 공개 화면은 법률사무소 홍보 페이지나 일반 뉴스 목록이 아니라, 사용자가 상황에서 출발해 법리ㆍ사실분기ㆍ공식 근거까지 이동하는 지식 인터페이스다.

따라서 시각적 완성도는 장식 이미지의 양보다 다음 순서를 눈으로 이해할 수 있는지로 판단한다.

1. 내가 겪은 상황
2. 한 줄 결론
3. 적용 법리
4. 결론을 가르는 사실
5. 공식 근거와 확인일
6. 개별 검토가 필요한 경계

## 2. 공통 시각 토큰

전역 스타일의 색상ㆍ표면ㆍ초점ㆍ그림자ㆍ모서리 값을 공통 토큰으로 관리한다.

- 녹색: 신뢰, 공식 근거, 주요 이동
- 청색: 구법ㆍ신법 비교와 법령변화
- 황색: 경과규정 미확인, 주의사항
- 흰색 표면: 독립된 법리ㆍ사실분기ㆍ콘텐츠 카드
- 크림색 바탕: 긴 글을 읽을 때의 눈부심 완화
- 공통 초점 표시: 키보드 사용자가 현재 위치를 확인하는 상태
- 공통 카드 그림자와 모서리: 콘텐츠 유형이 늘어나도 화면 밀도를 일관되게 유지

구체 값은 `app/globals.css`의 `:root`를 단일 기준으로 사용한다. 개별 화면에서 같은 의미의 색상이나 그림자를 임의로 다시 만들지 않는다.

## 3. 콘텐츠 유형의 시각 언어

지식 콘텐츠 유형 이름은 `src/lib/content-labels.ts`에서 한 번만 정의한다.

- 법령 변경
- 법리 해설
- 사실 분기
- 판례 법리
- 유사사례 비교
- 오해 바로잡기
- 절차와 증거
- 반복 쟁점

홈ㆍ지식 보관함ㆍ허브ㆍ통합 검색ㆍ상세 화면은 같은 유형 이름을 사용한다. 내부 식별자나 영문 유형값을 사용자 화면에 노출하지 않는다.

## 4. 카드와 화면의 역할

- 시작점 카드: 시간ㆍ사실ㆍ행동 중 사용자의 질문 방식을 선택한다.
- 법령변화 카드: 시행일과 구법ㆍ신법 차이를 강조한다.
- 지식 카드: 콘텐츠 유형, 대상 상황, 한 줄 결론을 먼저 보여준다.
- 법리카드: 누가ㆍ어떤 때ㆍ결과의 규범 슬롯을 보여준다.
- 사실분기 카드: 질문, 확인할 사실, 해당/비해당 결과를 나란히 보여준다.
- 공식 근거: 원문 주소와 마지막 확인일을 함께 표시한다.
- 컨시어지 연결: 공개 지식으로 결정할 수 없는 개별 사실만 별도 사이트로 넘긴다.

## 5. 접근성 기준

- 모든 링크ㆍ버튼ㆍ입력에는 키보드 초점 상태가 있어야 한다.
- 색상만으로 상태를 구분하지 않고 상태 이름을 함께 쓴다.
- 동작 감소 설정을 사용하는 기기에서는 이동 효과를 사실상 제거한다.
- 고대비 설정에서는 경계선과 핵심 링크를 강화한다.
- 모바일 제목은 한국어 단어 단위 가독성을 우선하고 긴 문자열만 안전하게 줄바꿈한다.
- 터치 화면에서 메뉴와 필터가 지나치게 작아지지 않게 한다.

## 6. 이미지와 별도 디자인 작업의 기준

현재 단계에서는 장식용 삽화나 사진 애셋을 대량 생성하지 않는다. 법률정보 사이트의 신뢰도와 확장성에는 타이포그래피, 정보 위계, 상태 표현과 탐색 구조가 더 직접적으로 기여한다.

별도 시각 디자인 작업은 다음 산출물이 필요할 때 제한적으로 연다.

- 구법 → 신법 → 적용시점처럼 문장보다 도식이 명확한 법률 흐름도
- 여러 사실분기가 세 단계 이상 연결되는 판단 트리
- RuleLink 고유 상표, 파비콘, 공유 이미지
- 실제 사용자 화면을 기준으로 한 반응형 디자인 점검

생성한 애셋은 의미, 대체문구, 사용 위치, 원본과 최적화본을 함께 관리한다. 콘텐츠마다 분위기만 다른 장식 이미지를 만드는 방식은 사용하지 않는다.

## 7. 확장 규칙

새 콘텐츠 유형이나 화면을 추가할 때 다음 순서로 작업한다.

1. 기존 토큰과 카드 역할로 표현 가능한지 확인한다.
2. 공통 유형 이름과 메타데이터를 먼저 추가한다.
3. 홈ㆍ보관함ㆍ검색ㆍ허브ㆍ상세 화면에서 같은 표현을 사용한다.
4. 키보드ㆍ모바일ㆍ동작 감소ㆍ고대비 상태를 확인한다.
5. 실제 승인 번들의 모든 공개 경로를 런타임 검사한다.

이 원칙은 특정 시안에 화면을 고정하기 위한 것이 아니라, 콘텐츠가 늘어도 사용자가 같은 방식으로 읽고 판단할 수 있게 하는 확장 계약이다.

## 8. 별도 시각 작업 세션 운영 계약

시각 작업은 상시 독립 개발팀이 아니라, 명확한 산출물을 가진 한시적 전담 세션으로 운영한다. 공개 웹 세션이 제품 구조와 최종 통합을 맡고, 시각 세션은 기존 공개 데이터 계약을 소비하는 화면 계층만 개선한다.

시각 세션의 필수 산출물은 다음과 같다.

1. 실제 공개 화면을 기준으로 한 시각 문제 목록과 개선 우선순위
2. 색상ㆍ서체ㆍ간격ㆍ표면ㆍ상태 표현을 재사용하는 토큰 변경
3. 홈ㆍ허브ㆍ검색ㆍ상세 화면에 공통 적용되는 컴포넌트
4. 모바일ㆍ키보드ㆍ고대비ㆍ동작 감소 상태의 확인 결과
5. 새 애셋이 필요한 경우 의미ㆍ대체문구ㆍ사용 위치ㆍ원본ㆍ최적화본을 적은 애셋 목록
6. 변경 전후 화면과 남은 제약을 담은 한글 검수 기록

역할 경계는 다음과 같다.

- 수정 가능: 공개 화면 컴포넌트, 전역 스타일, 공개 정적 애셋, 시각 체계 문서와 시각 회귀검사
- 수정 금지: 승인 번들 본문, 법리ㆍ사실분기ㆍ근거 좌표, 출판 스키마, 승인 상태와 최신성 기한
- 실제 Vercel 시각 검수가 필요하면 `preview-*` 브랜치를 사용한다.
- 문서 정리나 기계검사만 바꾸는 일반 브랜치는 Vercel 미리보기를 만들지 않는다.
- 최종 병합 전 공개 웹 세션이 정보 위계, 데이터 참조 무결성, 접근성, 정적 빌드와 전체 공개 경로를 다시 검증한다.

시각 세션은 콘텐츠별 장식 이미지를 먼저 만들지 않는다. 같은 법리와 상태가 모든 화면에서 같은 모양과 순서로 읽히게 만드는 것을 첫 번째 성공 기준으로 삼는다.
