# 공개 법률개념 용어 관계 계약

## 목적

본문의 법률용어 밑줄은 단순한 문자열 강조가 아니라 특정 개념카드로 연결되는 의미 관계다. 따라서 검색 편의를 위한 표현을 모두 같은 뜻으로 처리하지 않는다.

## 개념 하나의 기준

다음 중 하나라도 다르면 별도 개념카드를 만든다.

- 적용 대상이나 주체가 다르다.
- 성립 요건이 다르다.
- 법률효과가 다르다.
- 다른 개념의 상위·하위 개념이다.
- 같은 사람이라도 특정 관계나 절차에서의 지위를 가리킨다.

예를 들어 상속인, 법정상속인, 공동상속인은 관련성이 높지만 동일한 정의로 합치지 않는다.

## 검색 표현과 본문 자동 해설의 분리

- `aliases_ko`: 하나의 canonical concept(정본 개념 정체성)에 속하는 동의어·약어·표기변형·쉬운 표현만 둔다.
- 적용 주체·요건·효과가 다른 표현은 별칭으로 흡수하지 않고 별도 개념카드로 만든다.
- `term_relations`: 별칭의 관계 또는 별도 개념 사이의 의미 관계와 공식 근거를 기록한다.
- 본문 자동 해설은 `exact_synonym`, `abbreviation`, `spelling_variant`만 사용한다.
- `plain_language`, `narrower`, `broader`, `related`는 검색 연결에는 사용할 수 있지만 같은 팝오버를 띄우지 않는다.
- 관계 분류나 공식 근거가 없으면 본문에 밑줄을 만들지 않는다.

## canonical concept 정체성

- 대표 용어와 별칭은 NFKC(유니코드 호환 정규화)·대소문자·연속 공백을 정규화한 정확 일치 키로 전역에서 한 개념만 소유한다.
- 상속인·법정상속인·공동상속인·피상속인은 각각 별도 정체성이다. 긴 용어 안에 `상속인`이 들어간다는 이유로 별칭 처리하지 않는다.
- 임금·급여·보수·퇴직금은 문맥상 함께 검색될 수 있어도 하나의 정체성으로 무분별하게 합치지 않는다.
- 통상임금과 평균임금은 요건과 사용처가 다른 별도 정체성이다.
- `3.3%`와 `사업자등록`은 근로자성 판단자료일 수 있지만 근로자성의 별칭이 아니다.
- `원본`처럼 문서 원본과 채무 원본 등 여러 뜻을 가진 짧은 표현은 전역 별칭으로 금지한다. 필요한 경우 `채무의 원금(법률상 원본)`처럼 법역과 뜻을 대표 용어 자체에 한정한다.

## 개념 사이 관계

- `narrower`, `broader`, `related`는 `target_concept_id`로 별도 개념카드를 정확히 가리킨다.
- `term_ko`는 대상 개념의 대표 용어와 정확히 일치해야 하며 대상 개념이 실제로 존재해야 한다.
- 자기참조는 금지한다.
- `narrower`의 역방향은 `broader`, `broader`의 역방향은 `narrower`, `related`의 역방향은 `related`로 함께 선언한다.
- `narrower`·`broader`를 하나의 방향 그래프로 환산했을 때 순환이 있으면 개념 계층을 확정할 수 없으므로 출판을 차단한다.

## 본문 일치와 한 토큰 한 해설

- 본문 자동 해설은 정확한 문자열 가운데 가장 긴 용어를 먼저 선택한다.
- 짧은 대표 용어가 더 긴 복합 법률용어 내부에 포함됐다는 이유로 부분 일치시키지 않는다.
- 한 텍스트 토큰은 한 번만 분할하고 하나의 개념 팝오버만 연결한다. 같은 범위에 밑줄이나 버튼을 중첩하지 않는다.

## 기존 스냅샷의 품질부채

스냅샷 022의 `concept.kr.inheritance.legal_heir`가 `상속인`을 대표 용어로 두고 `법정상속인·공동상속인`을 별칭으로 둔 상태는 올바른 모델이 아니다. 다만 이 PR은 출판 정본을 수정하지 않으므로 해당 카드의 전체 내용 해시 영수증과 예상 오류 2건이 모두 정확히 일치할 때만 legacy debt(기존 품질부채)로 감사 통과시킨다. 카드가 조금이라도 수정되거나 다음 스냅샷으로 넘어가면 같은 예외를 재사용할 수 없고 엄격 검증을 통과해야 한다.

## 생산 절차

1. 주제 콘텐츠의 법리·사실분기·질문·검색어에서 반복되는 어려운 표현을 수집한다.
2. 활성 룰링크 원장과 공식 근거 좌표로 의미와 사용 문맥을 확인한다.
3. 기존 개념과 완전히 같은지, 쉬운 표현인지, 상·하위 또는 관련 개념인지 분류한다.
4. 요건·효과·한계가 다르면 새 개념카드를 작성한다.
5. 별칭에는 동의·표기 관계를, 별도 개념 관계에는 `target_concept_id`와 역방향 관계를 기록한다.
6. 출판 조립기가 근거 누락, 관계 미분류, 중복 자동 해설 용어를 차단한다.
7. 승인된 개념만 본문 팝오버·개념 페이지·검색에 반영한다.

## 축적 우선순위

다음 점수를 기계적으로 집계해 높은 표현부터 개념카드로 만든다.

- 공개 콘텐츠에서 등장한 서로 다른 글의 수
- 같은 글 안의 반복 횟수
- 법리 결론을 이해하는 데 필요한 정도
- 일반어와 법률상 의미가 달라 오해하기 쉬운 정도
- 이미 연결된 공식 근거 좌표의 유무

최종 정의와 관계 분류는 법률 담당자가 확인하되, 후보의 발견과 우선순위는 위 기준으로 반복 가능하게 산출한다.

## 금지사항

- 긴 표현이 짧은 대표 용어를 포함한다는 이유만으로 별칭 처리하지 않는다.
- 검색 유입을 늘리기 위해 근거 없는 동의어를 추가하지 않는다.
- 별도 개념이 필요한 표현을 기존 팝오버로 임시 연결하지 않는다.
- 팝오버가 없는 표현을 오류로 보지 않는다. 잘못된 설명보다 미표시가 우선이다.
