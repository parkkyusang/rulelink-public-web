# Vercel 배포 정책

## 결론

룰링크 공개 웹은 GitHub Actions를 변경 검증 계층으로 사용하고, Vercel은 프로덕션 배포와 의도적인 시각 검수에만 사용한다.

- `main`: 자동 프로덕션 배포
- `preview-*`: 시각 검수 목적의 명시적 미리보기 배포
- 그 밖의 모든 브랜치: Vercel 자동배포 차단

## 이유

콘텐츠 브랜치마다 Vercel 미리보기를 만들고 곧바로 `main`을 다시 빌드하면, 무료 플랜의 빌드 빈도 제한을 소모해 정작 프로덕션 배포가 거절될 수 있다. 룰링크 공개 웹은 이미 PR마다 다음 항목을 GitHub Actions에서 검증한다.

- 공개 번들 안전 규칙
- 주제 원본과 현재 공개 묶음의 결정론적 합성
- 불변 스냅샷과 해시 영수증
- 타입 검사
- 프로덕션 정적 빌드
- 공개 경로 런타임 스모크 테스트

일반 콘텐츠 PR은 이 검증으로 충분하다. 실제 브라우저 시각 검수가 필요한 디자인 변경만 `preview-*` 브랜치에서 Vercel 미리보기를 사용한다.

## 설정 계약

프로젝트 루트의 `vercel.json`은 다음 의미를 갖는다.

- `"**": false`: 이름에 `/`가 포함된 브랜치까지 모든 자동배포를 기본 차단
- `"main": true`: 프로덕션 브랜치는 예외적으로 허용
- `"preview-*": true`: 명시적 시각 검수 브랜치는 예외적으로 허용

Vercel은 `minimatch` 문법으로 브랜치 이름을 판정한다. `*`는 `/`를 넘지 못하므로 `codex/update-name` 같은 브랜치를 차단하지 못한다. 전체 기본 차단에는 반드시 `**`를 사용한다. 여러 패턴이 한 브랜치에 함께 일치하면 하나라도 `true`인 규칙이 있을 때 배포하므로 `main`과 `preview-*`만 예외적으로 배포된다.

## 작업 규칙

1. 콘텐츠·검색·데이터 구조 변경은 `codex/*` 브랜치에서 작업한다.
2. GitHub Actions가 모두 통과하면 `main`에 병합한다.
3. 화면의 색상·간격·반응형 배치처럼 실제 브라우저 확인이 필요한 변경은 `preview-*` 브랜치를 사용한다.
4. 미리보기 검수를 마친 뒤 동일 커밋을 PR로 병합하고, `main` 프로덕션 배포를 확인한다.
5. 이 정책을 바꿀 때에는 `scripts/vercel-deployment-policy.test.mjs`도 함께 갱신한다.
6. 전체 차단 규칙을 `*`로 축소하지 않는다. `/`가 들어간 작업 브랜치가 자동배포 대상으로 되돌아가기 때문이다.

## 실패 시 확인 순서

1. GitHub `main`의 최신 커밋을 확인한다.
2. 해당 커밋의 Vercel 상태가 `pending`, `success`, `failure` 중 무엇인지 확인한다.
3. 실패 주소가 `build-rate-limit` 또는 일일 배포 한도이면 코드 실패와 구분한다.
4. 설정 변경으로 새 미리보기 빌드가 불필요하게 발생하지 않았는지 확인한다.
5. 프로덕션 성공 뒤 `publication.json`의 `snapshot_id`와 공개 경로를 대조한다.
